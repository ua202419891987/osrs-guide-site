Windrose Dedicated Server Port 3478 Forward Fix

You set up a Windrose dedicated co-op server, the process is running, but your friends see "Failed to join" or time out forever. Nine times out of ten the server's UDP port 3478 isn't reachable from the internet — it's blocked by the server's router NAT or the machine's firewall. This host-side guide walks you through forwarding 3478 and opening it end to end.

🖥️ Level: Intermediate · 10 min read · Updated July 2026

⏱️ Quick Summary — 30-Second Read

  • 📌 Symptom: server runs, but remote players can't join — local/LAN players may work while internet players can't.
  • 📌 Root cause: UDP port 3478 isn't forwarded on the server's router or opened in the server firewall.
  • 📌 Step 1: forward UDP 3478 from the router to the server's LAN IP.
  • 📌 Step 2: allow UDP 3478 inbound in the Windows/Linux firewall.
  • 📌 Step 3: verify from outside with an online UDP port checker or a friend.

1. 🖥️ What This Fix Is For

This is the host / server operator counterpart to the player-side guide. If you run a dedicated Windrose co-op server — a separate PC, a home lab box, or a rented machine behind your own router — then you are responsible for making UDP 3478 reachable from the public internet. The fix lives on your network and your server OS, not on each player's machine.

⚠️ LAN Works, Internet Doesn't?

If players on the same network as the server can join but remote friends cannot, that's the textbook sign of a missing router forward / firewall block. LAN traffic never leaves your router, so it bypasses the exact rule that's failing for the internet.

A clean server setup needs three things lined up: the server process actually listening on 3478, the firewall permitting it, and the router sending external UDP 3478 to the server's internal IP. Miss any one and remote joins fail.

2. 📡 Understanding Port 3478 in Windrose Co-op

Windrose dedicated servers advertise and exchange real-time session data over UDP port 3478. When a player clicks "Join," their client sends a UDP packet to your public IP on 3478. The path it must travel:

StepWhat must happenFails if…
1. Internet → RouterRouter receives UDP 3478 on its public IPNo port-forward rule exists
2. Router → Server LAN IPNAT maps 3478 to the server's internal addressWrong internal IP / port
3. OS FirewallWindows/Linux allows inbound UDP 3478Firewall blocks the port
4. Server ProcessWindrose server is listening on 3478Server not running / wrong port
💡 UDP, Not TCP

Set every rule to UDP. A TCP-only forward on 3478 will look "open" in some checkers but Windrose co-op packets will still be dropped. When in doubt, choose "UDP" or "Both."

3. 🔧 Forward Port 3478 on the Server Router

This is the single most important step. You're telling the router: "anything that arrives on public UDP 3478 belongs to the server at this LAN address."

① Assign the Server a Static LAN IP

Reserve a fixed address for the server (e.g. 192.168.1.20) via DHCP reservation in the router, or set it statically on the server OS. A changing IP would break the forward every reboot.

② Open Port Forwarding on the Router

Log in to the router (often 192.168.0.1 / 192.168.1.1). Go to Port Forwarding / Virtual Server / NAT and add:

FieldValue
NameWindrose-Server
ProtocolUDP
External / WAN port3478
Internal / LAN port3478
Internal IPserver static LAN IP (e.g. 192.168.1.20)
StatusEnabled

③ Save & Reboot the Router

Apply the rule and reboot the router so NAT reloads cleanly. Confirm the server still has the reserved IP before testing.

⚠️ Double NAT Trap

If your server sits behind two routers (e.g., ISP modem/router + your own), you must forward 3478 on both, or put the ISP box in bridge mode. A single forward on the inner router won't help if the outer one still drops 3478.

4. 🧱 Open Port 3478 in the Server Firewall

Even with a perfect router forward, the server's own OS firewall can silently drop inbound UDP 3478. Open it on whatever OS your server runs.

🪟 Windows Server / PC

  1. Open Windows Defender Firewall → Advanced Settings.
  2. Select Inbound Rules → New Rule → Port.
  3. Choose UDP, Specific local ports: 3478.
  4. Action: Allow the connection.
  5. Apply to Domain / Private / Public (or at least the network profile the server uses).
  6. Name it "Windrose Co-op UDP 3478" and finish.

Or run as admin in PowerShell:

New-NetFirewallRule -DisplayName "Windrose 3478 UDP" -Direction Inbound -Protocol UDP -LocalPort 3478 -Action Allow

🐧 Linux Server (ufw / iptables)

  1. With ufw: sudo ufw allow 3478/udp then sudo ufw reload.
  2. With raw iptables: sudo iptables -A INPUT -p udp --dport 3478 -j ACCEPT.
  3. On systemd servers, also confirm no firewalld zone is blocking: sudo firewall-cmd --add-port=3478/udp --permanent && sudo firewall-cmd --reload.
🔔 Cloud / VPS Hosts

If your "dedicated server" is a VPS (AWS, GCP, Azure, a game host), you usually also have a provider security group / firewall layered on top. Open inbound UDP 3478 there too — the OS rule alone won't help if the platform blocks it.

5. 🧪 Verify the Port Is Reachable

Don't assume — verify from outside your network. A single missed step still blocks joins.

① Confirm the Server Is Listening

On the server, check the process is bound to 3478:

# Linux: ss -lunp | grep 3478   # Windows: netstat -an | find "3478"

You should see 0.0.0.0:3478 (or :3478) in a LISTENING / bound state.

② Test From a Remote Location

From a different network (a friend's house, a phone on cellular, or a cloud shell), use an online UDP port checker against your public IP on 3478 — or simply have a remote friend try to join. Success = the full chain works.

③ Check Your Public IP Matches the Forward

If your ISP uses CGNAT, your "public" IP shown by the router may not be the real internet-facing one, so the forward points at nothing reachable. Compare the router's WAN IP with a site like "whatismyip" — if they differ, you're behind CGNAT (see pitfalls).

⚠️ UDP Checkers Can Be Tricky

Because UDP is connectionless, some online checkers report "closed" even when the server answers. The most reliable test is a real remote friend joining while the server is live.

6. 🚨 Common Host-Side Pitfalls

  • Double NAT: two routers in line — forward on both or bridge the ISP modem.
  • CGNAT: ISP shares one IP — request a public IP, or run the server behind a port-forwarding VPN / tunnel.
  • Wrong protocol: TCP-only forward — must be UDP.
  • Dynamic server IP: DHCP reassigned the server's LAN IP — use a static reservation.
  • Firewall on the host OS: router is fine but Windows/Linux still blocks — open UDP 3478 inbound.
  • Server not actually on 3478: a config flag set a different port — match the forward to the real listen port.
💡 Tell Players to Read the Player Fix Too

Even with a perfect server, a joining player behind a Strict NAT may still struggle. Point them to the player-side guide so both ends are clean — co-op needs at least one side with a usable NAT.

7. ❓ FAQs

Do I need to forward 3478 if players join over a lobby code?

Yes — relaying still ultimately needs the server reachable on UDP 3478 from the internet. The lobby code doesn't bypass the port; it just tells clients where to send the 3478 packets.

TCP or UDP for the server forward?

UDP. Windrose co-op uses UDP 3478. A TCP-only rule will not let players join.

LAN players join but internet players can't — what's wrong?

Classic missing router forward or firewall block. LAN traffic skips NAT/firewall routing, so it works locally while remote 3478 traffic is dropped. Follow sections 3–4.

My router WAN IP differs from whatismyip — now what?

You're behind CGNAT. Ask your ISP for a public IPv4, or host the server behind a VPN/tunnel that supports UDP port forwarding.

Does the server need UPnP instead of manual forwarding?

Manual forwarding is more reliable for a dedicated server, especially on Linux/headless boxes where UPnP may be off. Set the static forward and firewall rule directly.
