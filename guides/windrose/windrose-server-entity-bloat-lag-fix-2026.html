Windrose Server Entity Bloat Causing Lag — Fix

🏴‍☠️ Intermediate · Server / Performance Fix · 9 min read · Updated July 2026

Your Windrose dedicated or co-op server started smooth, then after a few in-game weeks it began to stutter, rubber-band, and drop ticks. The usual culprit is not your hardware — it is entity bloat: thousands of un-despawned objects (dropped loot, corpses, player structures, dormant NPCs, and stray projectiles) piling up on the server until the simulation can no longer keep up. This guide explains the mechanics and gives you a repeatable cleanup routine.

⏱️ Quick Summary — 30-Second Read

  • 📌 Symptom: server runs fine for days, then ticks drop, players rubber-band, and the simulation slows to a crawl.
  • 📌 Cause: entities that never despawn (loot, corpses, structures, dormant NPCs) accumulate until the server is overloaded.
  • 📌 Fix 1: learn the despawn timers and stop leaving loot on the ground.
  • 📌 Fix 2: cap your structures per island and consolidate storage.
  • 📌 Fix 3: run periodic admin cleanup and schedule automatic restarts.

1. 🌊 What Entity Bloat Looks Like

Entity bloat is slow and sneaky. A fresh Windrose server simulates a few hundred objects comfortably — ships, players, fish, weather, a handful of NPCs. Over time, every dropped cannonball, every uncollected treasure pile, every sunk-ship wreck, every decorative chest, and every despawned enemy corpse adds one more object the server must track every tick. None of these are visible as "lag" individually, but together they push the server past its comfortable ceiling.

The classic progression is: week one feels buttery smooth; week two you notice occasional micro-stutters during combat; week three the server ticks start dropping during peak hours and players complain about rubber-banding; week four the world literally crawls — cannons fire in slow motion, NPCs teleport, and the save takes forever. That curve is the fingerprint of entity bloat, not a bad GPU or a weak internet connection.

⚠️ It Is the Server, Not the Client

If every player on the server lags at the same time — regardless of their own PC or console — the bottleneck is the simulation running on the host. Client-side fixes (lower graphics, reinstall) will not help. You need to reduce what the server has to simulate.

2. 🧠 Why Servers Get Bloated

Windrose tracks "entities" on the server as long-lived simulation objects. Some are supposed to stay forever (your built base, your anchored ship). Others are meant to be temporary — but only if their despawn condition is actually met. Bloat happens when temporary entities are created faster than they are removed, or when "permanent" entities multiply without limit.

The biggest sources, in practice:

  • Dropped loot: players toss junk on the deck or shore instead of selling or storing it. Each pile is a tracked entity.
  • Corpses & wrecks: sunk enemy ships and defeated enemies leave bodies that linger.
  • Redundant structures: dozens of chests, barrels, and decorative blocks that could be consolidated into a few storage units.
  • Dormant NPCs: villagers, merchants, and wildlife that have wandered into loaded areas and never leave.
  • Stray projectiles & effects: uncollected cannonballs, lingering status effects, and orphaned physics objects after a crash.

On a non-dedicated (host-played) session the world unloads when the host sails away, which naturally clears a lot of this. On a dedicated server that runs 24/7, nothing ever unloads on its own — so bloat compounds until you intervene. If you run a persistent world, the routine below is mandatory, not optional. 3. ⏳ Despawn Rules You Should Know

Windrose does have automatic despawn logic, but the timers vary by entity type — and several objects are exempt entirely. Knowing the rules lets you stop bloat at the source instead of cleaning it up later. The table below summarizes the typical behavior (exact values can shift between patches, so treat these as planning targets rather than guarantees).

Entity typeTypical despawnAuto-clears?Notes
Loose dropped items~10–20 minYesLongest if on a loaded chunk; pick up loot promptly.
Enemy corpses~5–10 minYesCan linger after large battles; clear the field.
Sunk-ship wrecks~15–30 minUsuallyBig wreck models are heavy; sink them away from base.
Dormant wildlife/NPCNone (persists)NoStay loaded if near a player or build.
Player structuresNeverNoPermanent until demolished by a player or admin.
Anchored shipsNeverNoCount toward the server's persistent entity load.
Stray projectiles~1–3 minYesOrphaned after crashes; restart clears them.
💡 The Key Insight

Only the temporary rows above self-clean. Everything marked "No" stays until a human or an admin command removes it. Bloat is almost always the accumulation of the "No" rows — structures and dormant NPCs — plus temporary rows that never got a chance to despawn because the chunk stayed loaded.

4. 🏗️ Limit Structures & Builds

Structures never despawn, so they are the part of the entity count you control most directly. A tidy, intentional base is dramatically cheaper to simulate than a sprawling one. You do not need to live in a shoebox — you need to be deliberate.

CategoryRecommended approachWhy it helps
StorageUse 1–2 large chests per function, not 20 small onesFewer container entities, same capacity
DecorationsCap non-functional blocks per islandPurely cosmetic entities still cost ticks
Ships anchoredAnchor only what you use; un-anchor the restEach anchored hull is a persistent load
AutomationAvoid redundant duplicated stationsCut passive simulation overhead
OutpostsConsolidate into one main base when possibleFewer loaded build clusters

① Audit Your Base Monthly

Walk every island you own and demolish anything you do not actively use. Old test builds, duplicate chests, and "someday" projects are the silent killers of server performance.

② Consolidate Storage

Twenty small chests = twenty entities. Two large chests = two entities with the same total space. Consolidation is the single highest-impact change most bloated servers can make.

③ Keep Combat Away From Base

Sink enemy ships and fight battles at a distance. Wrecks and corpses that spawn inside your permanently-loaded base radius will sit there until manually cleared.

5. 🧹 Admin Cleanup Routine

If bloat has already set in, the fastest fix is an admin cleanup pass. Windrose dedicated servers expose admin commands that let you wipe loose entities, clear dropped items, and remove dormant NPCs without touching player-built structures. The exact command syntax lives in our admin reference — consult it for the precise tokens on your build.

📋 A Safe Cleanup Pass (order matters)

  1. Warn players — announce a 5-minute cleanup window so nobody loses an item they just dropped.
  2. Clear dropped items — purge loose loot and stray projectiles from the world.
  3. Remove dormant NPCs — despawn wildlife/merchants that drifted into loaded zones.
  4. Clear corpses & wrecks — remove lingering battle debris.
  5. Verify structures — do NOT mass-demolish; only remove orphaned/duplicate builds with player consent.
  6. Save and report — confirm the entity count dropped and tell players the server is healthy again.
⚠️ Protect Player Bases

Admin "wipe all" commands exist — and they will delete player-built structures and stored items. Never run a blanket wipe to fix lag. Target only loose entities and dormant NPCs. If you need the full command list and safe flags, see the dedicated admin reference and our Performance Optimization Guide 2026.

💡 Automate It

Once you know the safe commands, script a lightweight cleanup that runs on a schedule (e.g., nightly) so loose entities never pile up. Pair it with the restart routine in the next section for a self-maintaining server.

6. 🔄 Restart the Server

A clean restart is the cheapest, most reliable bloat remedy — and it clears the categories nothing else touches. When the server process exits and reloads, transient memory (orphaned projectiles, stuck physics objects, desync states, and un-collected effects from a crash) is released. Persistent structures and stored items remain, because they live in the save file, not in volatile memory.

① Schedule Regular Restarts

For a 24/7 dedicated server, a daily or every-other-day restart during low-traffic hours prevents slow accumulation. Most server hosts let you set a cron-style restart schedule — use it.

② Save Before You Restart

Always trigger a manual save (or confirm autosave fired) immediately before restarting. A restart mid-unsaved-state can roll back recent progress and, ironically, leave orphaned entities behind.

③ Restart, Don't Just Reload the Map

A full process restart clears volatile memory. A soft "reload" that keeps the process alive may keep the bloat. Confirm the server process actually stopped and started fresh.

🔔 Combine Restart + Cleanup

The most robust setup: a nightly admin cleanup pass, followed by an automatic restart. Together they keep entity counts low permanently with zero manual effort after the first configuration.

7. ❓ FAQs

Why does lag only appear on my dedicated server, not in solo play?

In solo play the world unloads when you sail away, which silently clears most temporary entities. A dedicated server runs continuously, so nothing unloads on its own — bloat accumulates until you intervene with cleanup and restarts.

Do dropped items really cause noticeable lag?

Individually, no. But a busy server can accumulate hundreds of loose item stacks and stray projectiles over weeks. Each is a tracked entity the server simulates every tick, and the combined load is a major bloat source. Pick up loot and run cleanup passes.

How often should I restart the server?

For a persistent 24/7 world, a restart every 1–2 days during off-peak hours is a good baseline. If you also run scheduled admin cleanup, you can stretch toward every 2–3 days. Watch tick rate; if it dips, restart sooner.

Will an admin cleanup wipe my base or my friend's storage?

Not if you target only loose entities and dormant NPCs. Structures and stored items are permanent and separate from the cleanup commands. Avoid any "wipe all" or "demolish everything" command — those are the ones that delete bases.

Can a non-dedicated co-op session get entity bloat?

Less often, because the host sailing away unloads the area. But long co-op sessions in one persistent region — especially with many structures — can still bloat. A session restart and a tidy base (see the Beginner Guide 2026) cover most cases.
